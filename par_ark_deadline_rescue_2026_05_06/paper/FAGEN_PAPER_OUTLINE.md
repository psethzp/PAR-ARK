## Title

When Adaptive Knowledge-Graph Retrieval Fails: Trace Diagnostics for Local LLM Agents on STaRK

## Abstract draft

Adaptive graph retrieval agents promise to balance broad global search with local multi-hop exploration, but their failures are difficult to diagnose from terminal ranking metrics alone. We study a local open-weight ARK-style retrieval agent on STaRK using Qwen3-14B and Qwen3-30B-A3B controllers. We add profile-conditioned and trace-conditioned retrieval hints intended to improve breadth-depth control. Instead, across validation and pilot test settings, the full trace/profile intervention consistently reduces Hit@1, Hit@5, Recall@20, and MRR relative to the local baseline, despite often reducing wall-clock time. We show that trace metrics explain the failure: the intervention increases zero-result neighborhood calls, repeated tool calls, and over-exploration before answer commitment. Our results provide a reproducible failure trigger for adaptive KG retrieval agents, a trace-level diagnostic protocol, and budget-constrained mitigation probes that quantify the cost of safer control. The study argues that agentic retrieval systems should be evaluated by both final ranking quality and process-level tool traces.

## 1. Introduction

Problem:

- KG retrieval agents are attractive because they can reason over graph structure at inference time.
- Final retrieval metrics only show whether an answer was ranked correctly; they hide the process failure that caused the ranking loss.
- Local open-weight agents are especially relevant because many users cannot use closed API models.

Thesis:

- A seemingly sensible profile/trace controller can make a local retrieval agent worse.
- This is not a failed experiment to hide; it is exactly the kind of reproducible agent failure that should be reported.

Contributions:

1. A reproducible failure trigger for ARK-style adaptive retrieval on STaRK: enabling profile/trace hints.
2. A trace diagnostic protocol: global calls, neighborhood calls, zero-result calls, repeated calls, step count, time, and paired rank deltas.
3. A budget-fix probe showing whether constraining trace actions mitigates harm and what it costs.
4. Open local-model evidence with Qwen3-14B and Qwen3-30B-A3B.

## 2. Background

STaRK benchmark:

- Three domains: Amazon/products, MAG/academic, PRIME/biomedical.
- Metrics: Hit@1, Hit@5, Recall@20, MRR.

ARK baseline:

- Global BM25 search + one-hop neighborhood exploration.
- Strong closed-model reference.

Local setup:

- Qwen3-14B validation grid.
- Qwen3-30B-A3B TP2 pilot/subset.
- No closed GPT API.

## 3. Intervention and Failure Hypothesis

Intervention:

- `profile`: prepend a structured query/profile block.
- `trace`: add trace hints after tool observations and budget events.
- `full`: profile + trace.

Intended effect:

- Improve adaptive breadth-depth retrieval.

Observed failure hypothesis:

- The local model over-trusts control hints, performs more neighborhood exploration, repeats failing calls, and commits fewer/bad answer nodes.
- Faster runtime is caused by premature bad termination, not better retrieval.

## 5. Discussion

Main lesson:

- Adaptive retrieval control is model-dependent. Closed-model ARK behavior does not transfer automatically to local open-weight controllers.
- Prompt-level control hints can make a local model spend budget on process actions that look rational but reduce rank quality.
- Trace diagnostics are necessary because final rank metrics do not reveal whether the problem is anchoring, over-exploration, repeated calls, or premature stopping.

## 6. Limitations

- The work is a deadline-constrained diagnostic study, not a new SOTA method.
- The local intervention is prompt/controller-level and does not train the model.
- Test subset may be smaller than the full split if Stage E does not complete.
- Persona/personalization is represented as profile conditioning, not a full user-memory benchmark.

## 7. Conclusion

We report a reproducible failure mode of adaptive KG retrieval agents under local open-weight models. The result is negative in accuracy but positive in diagnostic value: process traces explain why adaptive control harms retrieval and define concrete targets for future repair.
